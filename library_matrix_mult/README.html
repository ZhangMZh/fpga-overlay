Description

This example demonstrates testing and packaging multiple, interdependent libraries for use accross languages. Specifically we will work with HLS libraries for use from an OpenCL kernel. Each library contains its own testbench.

The libraries consist of a dot product library, and a matrix multiplication library that depends on it. Both are written in HLS C++. Note that this design is not optimized. For example, the OpenCL code uses a single work-item for simplicity.

The kernel file is in the device directory. The libraries are in device/lib and their OpenCL headers in device/inc. Take a look at the .cpp files in device/lib. Notice they contain a testbench main function, guarded by ifdef directives. We will test and package each library at different stages, mimicking a library development process.

Testing the Libraries

We will cover the packaging and testing of the libraries. All of the files needed for testing can be created by going to device/lib and running:

make test_dot_prod
make test_matrix_mult

First we will test the dot product library found in device/lib/dot_prod.cpp. The flag -DTEST_HLS_DOT_PROD controls the inclusion of the testbench main function for this library in compilation. Compile the HLS code using the simulation flow.

i++ -DTEST_HLS_DOT_PROD -march="Arria10" -I device/lib/ device/lib/dot_prod.cpp -o bin/test/dot_prod

This will create an executable for simulation. Run the testbench for dot_prod.

./bin/test/dot_prod

If all goes well, you should see some test output ending in PASSED.

Next we will test the matrix multiplication library which depends on dot product.

To compile the dot product code to an object file dot_prod.o, we will use the fpga_crossgen tool. Then to package the object file in a library archive file dot_prod.a, we will use fpga_libtool. Since we want to use the library from matrix_mult.cpp which is written in HLS, we will use the --target flag with option hls.

fpga_crossgen --target hls device/lib/dot_prod.cpp -o bin/test/dot_prod.o
fpga_libtool --target hls bin/test/dot_prod.o --create bin/test/dot_prod.a

Now we can use the dot product library when compiling our matrix multiplication code. The -DTEST_HLS_MATRIX_MULT flag controls the inclusion of the testbench for the matrix_mult library in compilation.

i++ -DTEST_HLS_MATRIX_MULT -march="Arria10" device/lib/matrix_mult.cpp bin/test/dot_prod.a -o bin/test/matrix_mult

Run the testbench for matrix multiplication and ensure that it passes.

./bin/test/matrix_mult

Packaging the Libraries for OpenCL

Now that we are confident in our HLS library designs, let's compile and package them to be used from an OpenCL kernel. As before we will use the fpga_crossgen and fpga_libtool tools. To indicate we want to use these libraries from OpenCL we will use the --target flag with option aoc. The flag -DTARGET_OCL controls the inclusion of a necessary OpenCL header in device/lib/dot_prod.h. Lastly, when targeting OpenCL, object files must use the extension .aoco and library files must use .aoclib.

fpga_crossgen --target aoc -DTARGET_OCL device/lib/dot_prod.cpp -o bin/dot_prod.aoco
fpga_crossgen --target aoc -DTARGET_OCL device/lib/matrix_mult.cpp -o bin/matrix_mult.aoco
fpga_libtool --target aoc bin/dot_prod.aoco --create bin/dot_prod.aoclib
fpga_libtool --target aoc bin/matrix_mult.aoco --create bin/matrix_mult.aoclib

These can be run by going to device/lib and running:

make all

Compiling the OpenCL Kernel

The top-level OpenCL kernel file is device/kernel.cl.

To compile the OpenCL kernel, run:

aoc device/kernel.cl -o bin/kernel.aocx -I device/inc/ -L bin/ -l matrix_mult.aoclib -l dot_prod.aoclib -board=<board>

where <board> matches the board you want to target. The -o bin/kernel.aocx argument is used to place the compiled binary in the location that the host program expects.

If you are unsure of the boards available, use the following command to list available boards:

aoc -list-boards

Compiling for Emulator

To use the emulation flow, the compilation command just needs to be modified slightly:

aoc -march=emulator device/kernel.cl -o bin/kernel.aocx -I device/inc/ -L bin/ -l matrix_mult.aoclib -l dot_prod.aoclib

Compiling for Simulator

To use the simulation flow, the compilation command just needs to be modified slightly:

aoc -march=simulator device/kernel.cl -o bin/kernel.aocx -I device/inc/ -L bin/ -l matrix_mult.aoclib -l dot_prod.aoclib -board=<board>

Compiling the Host Program

To compile the host program, run:

make

The compiled host program will be located at bin/host.

Running the Host Program

Before running the host program, you should have compiled the OpenCL kernel and the host program. Refer to the above sections if you have not completed those steps.

To run the host program on hardware, execute:

bin/host

Running with the Emulator

Prior to running the emulation flow, ensure that you have compiled the kernel for emulation. Refer to the above sections if you have not done so. Also, please set up your environment for emulation. Please see the Intel® FPGA SDK for OpenCL™ Programming Guide for more information.

For this example design, the suggested emulation command is:

bin/host

Running with the Simulator

Prior to running the Simulation flow, ensure that you have compiled the kernel for simulation. Refer to the above sections if you have not done so. Also, please set up your environment for simulation. Please see the Intel® FPGA SDK for OpenCL™ Programming Guide for more information.

For this example design, the suggested simulation command is:

CL_CONTEXT_MPSIM_DEVICE_INTELFPGA=1 bin/host

Host Parameters

The general command-line for the host program is:

bin/host [-n1=<uint>] [-n2=<uint>] [-n3=<uint>]

where the parameters are:

Parameter Type Default Description
-n1=<uint> Optional 6 Number of rows of the first matrix
-n2=<uint> Optional 8 Number of columns of the first matrix, and rows of the second matrix
-n3=<uint> Optional 4 Number of columns of the second matrix

OpenCL Binary Selection

The host program requires a OpenCL binary (AOCX) file to run. For this example design, OpenCL binary files should be placed in the bin directory.

By default, the host program will look for a binary file in the following order (earlier pattern matches take priority):

  1. A file named kernel.aocx.
  2. A file named kernel_<board>_191.aocx, where <board> is the name of the board (as passed as the -board argument to aoc).
